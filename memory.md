# Design Playbook

This file is the source of truth for how we build this hub and connected webapps.
Use it as the first reference for AI and humans.

## AI quick brief (read first)
- Implement directly. Do not stop at planning unless asked.
- Keep the existing visual direction: dark, flat, clean, readable.
- Mobile is mandatory. Horizontal overflow is a bug.
- Preserve behavior when refactoring.
- Run build and tests before finalizing.
- Use `npm.cmd` on Windows when PowerShell blocks `npm.ps1`.
- Keep secrets in environment variables only.

## Hard constraints

### Platform and deployment
- Node.js 20+.
- App must listen on `process.env.PORT`.
- Keep `NODE_ENV=production` in hosted environments.
- Keep Coolify compatibility.
- Local-first development on Windows should keep working.

### Security and API behavior
- Auth-protected endpoints must return JSON 401 when unauthenticated.
- API endpoints must return JSON on errors, not HTML.
- Validate input on both client and server.
- Keep clear, actionable API error messages.

### Build and source of truth
- `public/*.tsx` and `public/modules/*.tsx` are source files.
- `public/*.js` and `public/modules/*.js` are generated by `scripts/build-client.mjs`.
- Generate client assets with `npm.cmd run build:client` before deploy.
- Do not treat generated output as source of truth.

## Design system rules (current hub standard)

### Visual direction
- Discord-like dark UI, but flatter and cleaner.
- Avoid glossy effects, heavy gradients, and over-bright surfaces.
- Avoid overly rounded corners.
- Keep contrast balanced and text easy to read.

### Typography
- Body text: Manrope.
- Headings and key labels: Sora.
- Keep hierarchy clear: short label, strong title, practical helper text.

### Core tokens
- Background: `#090d14`.
- Surfaces: `#111720`, `#171d27`.
- Borders: `#252c38`, `#303a4a`.
- Text: `#f4f6fb`, `#9ba3b3`, `#626b7a`.
- Accent: `#5b5cff`, `#7b61ff`.
- Status: green `#22c55e`, amber `#f59e0b`, danger `#ff6c86`.

### Iconography and branding
- Use symbol-style SVG icons with `currentColor` when possible.
- Do not use emoji as UI icons.
- Keep sidebar helper label as "Workspace".
- Brand badge pattern stays compact and consistent.

### Copy and UX text
- Keep copy short, practical, and human.
- Avoid AI-sounding or promotional phrasing.
- Labels should be concise and clear.
- Error states must explain what happened and what to do next.

## Responsive rules
- Mobile responsiveness is required for every page.
- Horizontal scrollbars in primary layouts are regressions.
- Avoid `overflow-x: auto` as default for navigation and key data views.
- Convert wide desktop tables to stacked card rows on small screens.
- Navigation must wrap or reflow on small screens.

## General development rules
- User expects end-to-end fixes in one flow.
- Prefer direct implementation over long planning loops.
- Keep code modular and maintainable.
- Keep dependencies minimal and current.
- Preserve existing behavior while refactoring.
- Restart server after backend route changes to avoid stale runtime confusion.

### Architecture-specific rules
- Settings architecture uses modules under `public/modules/`.
- Do not query DOM at module import time.
- Initialize DOM refs after DOM ready (`initDOMRefs()` pattern).
- Keep section behavior centralized in init orchestration and module handlers.
- News and other private pages should keep the shared shell pattern.

## Webapp integration contract (for connected hub apps)

### App registration contract
Each app entry should provide:
- `id` (stable, kebab-case)
- `name`
- `description`
- `url`
- `tone`
- `availability`

### URL and page contract
- In-repo app pages should follow `/apps/<id>/` with `public/apps/<id>/index.html`.
- Private pages should keep the shared workspace shell pattern.
- New pages must include mobile nav hooks when using the shared rail.

### Icon and asset contract
- Register app icons in dashboard icon mapping with a fallback glyph.
- Keep app assets under `public/assets/cvgs/`.
- Remove obsolete asset variants after migration.

### API contract for connected apps
- Return JSON in success and error paths.
- Return 401 JSON for unauthenticated access to protected routes.
- Never return HTML from API endpoints.

## Quality control gates (must pass)

### Build and test
- `npm.cmd run build:client` passes.
- `npm.cmd run test` passes.

### Functional smoke checks
- Sign in and sign out flows work.
- Home/dashboard loads app list and key widgets.
- Settings tabs and module actions work.
- News, support, and device info pages load correctly.

### Responsive and UX checks
- No horizontal overflow at mobile widths.
- Navigation and key actions remain usable on mobile.
- Text is concise and practical.
- No emoji used as UI iconography.

### Runtime checks
- Protected APIs return JSON 401 when logged out.
- API endpoints do not leak HTML in error paths.

## Known good patterns
- Settings modularization with lazy DOM initialization is stable.
- Shared shell consistency across private pages reduces regressions.
- Rebuilding broken HTML from clean structure is faster than patching malformed markup.
- Capacity and API checks should fail clearly and recover cleanly.

## Update protocol
- Update this file when a rule is stable across at least two real tasks.
- If a rule is temporary, keep it out of this file.
- Keep this file short, strict, and actionable.
